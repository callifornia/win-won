✅✅✅ Теорема CAP: ✅✅✅
        • C ~> однакові дані
        • A ~> система завжди відповідає
        • P ~> працює навіть при збоях мережі

        • C ~ УЗГОДЖЕНІСТЬ (consistency)
            • якщо ви змінили пароль, то на всіх серверах одразу буде новий пароль
            • всі клієнти бачать однакові дані в один і той самий момент як у класичній базі даних (ACID)
        • A ~ ДОСТУПНІСТЬ (availability)
            • сайт продовжує працювати навіть при збоях одного сервера
            • кожен запит отримує відповідь (навіть якщо дані можуть бути неактуальні)
        • P ~ СТІЙКІСТЬ ДО РОЗПОДІЛЕННЯ МЕРЕЖІ (Partition_tolerance)
            • дата-центри в різних країнах тимчасово втратили з’єднання, але система не зупинилась
            • система працює навіть якщо між серверами зник зв’язок


        • SQL БД            однакові дані +  дата центри           CP
        • Cassandra         один сервер   +  дата центри           AP
        • MongoDB           однакові дані +  дата центри           CP



✅✅✅ ACID и BASE ✅✅✅

    • це моделі транзакцій для баз даних, які визначають структуру даних і порядок роботи з ними в базі даних
    • у контексті баз даних транзакцією називають будь-яку операцію, яку база даних обробляє як єдину неподільну одиницю роботи
    • щоб БД залишалася узгодженою, транзакція має бути повністю завершена
    • переказуєте гроші з одного банківського рахунку на інший, необхідно відобразити ці зміни як на вашому рахунку, так і на рахунку отримувача


    • ACID - модель транзакций
        • у БД ACID пріоритет надається узгодженості (consistency) на шкоду доступності (availability)
        • якщо на будь-якому етапі транзакції виникає помилка, вся транзакція повністю скасовується

    • BASE - модель транзакций
        • натомість БД BASE надають пріоритет доступності на шкоду узгодженості
        • у разі невдалого завершення транзакції користувачі можуть тимчасово отримувати неузгоджені дані
        • узгодженість даних відновлюється через певний проміжок часу



✅✅✅  BASE - модель транзакций  ✅✅✅
    • базова доступність (Basically Available)
        • база даних має бути доступною для всіх користувачів одночасно в будь-який момент часу
        • користувачеві не потрібно чекати завершення інших транзакцій, перш ніж оновлювати запис
    • м’який стан (Soft State)
        • будь-які дані можуть перебувати в проміжних або тимчасових станах і в певний момент змінюватися «самі по собі»,
          без зовнішніх подій або надходження нових даних. Ця концепція відображає невизначений стан запису, який одночасно оновлюється кількома застосунками
        • наприклад, якщо користувачі редагують повідомлення в соціальній мережі, внесені ними зміни можуть бути не відразу видимими для інших користувачів
          Через деякий час система врахує всі внесені зміни, і повідомлення буде оновлено, хоча жоден користувач безпосередньо не ініціював це оновлення
    • Поступова узгодженість (Eventually Consistent)
        Запис досягне узгодженого стану не відразу, а після завершення всіх одночасних оновлень



✅✅✅  АСИД - модель транзакций  ✅✅✅
    Набір правил які забезпечують цілісність данних

            A — Atomicity:      або все або нічого
            C — Consistency:    не повинна порушувати constraints: PK, FK, CHECK, UNIQUE
            I — Isolation:      не повинні ламати одна одну
            D — Durability:     після COMMIT дані не губляться


    • атомарність  - гарантує що транзакция буде виконана повність або не буде виконана взагалі. Не допускаються проміжні стани
    • узгодженість - перевод бази даних з одого согласованого стану в інший согласований стан
                     Приклад: В рамках одної транзакції є три insert -> у випадку якщо один insert впав з помилкою то відкатуються два інші
    • ізоляція     - події, що відбуваються всередині транзакції, повинні бути приховані від інших транзакцій, що одночасно виконуються
                     при параллельном выполнении транзакций возможны следующие проблемы:

                    • уровень READ UNCOMMITTED
                          Транзакция 1                              Транзакция 2
                          SELECT f2 FROM tbl1 WHERE f1=1;
                          UPDATE tbl1 SET f2=f2+1 WHERE f1=1;
                                                                     SELECT f2 FROM tbl1 WHERE f1=1;
                          ROLLBACK WORK;
                      В транзакции 1 изменяется значение поля f2, а затем в транзакции 2 выбирается значение этого поля. После этого происходит
                      откат транзакции 1. В результате значение, полученное второй транзакцией, будет отличаться от значения, хранимого в базе данных

                    • уровень READ COMMITTED
                              Транзакция 1                                Транзакция 2
                              SELECT f2 FROM tbl1 WHERE f1=1;             SELECT f2 FROM tbl1 WHERE f1=1;
                              UPDATE tbl1 SET f2=f2+1 WHERE f1=1;
                              COMMIT;
                                                                          SELECT f2 FROM tbl1 WHERE f1=1;
                        В транзакции 2 выбирается значение поля f2, затем в транзакции 1 изменяется значение поля f2. При повторной попытке выбора
                      значения из поля f2 в транзакции 2 будет получен другой результат. Эта ситуация особенно неприемлема, когда данные
                      считываются с целью их частичного изменения и обратной записи в базу данных.

                    • фантомное чтение (уровень READ REPEATABLE) — при повторном чтении в рамках одной транзакции прочитаны данные(новые "фантомные" строки),
                      которых при предыдущих чтениях не было. Попросту говоря, фантомное чтение может происходить в случае, если в одной транзакции выбирается некоторый диапазон строк,
                      затем другая транзакция вставляет новую строку в этот диапазон, после чего в первой транзакции выбирается тот же диапазон снова.
                      В результате первая транзакция увидит новую «фантомную» строку.
                              Транзакция 1                                  Транзакция 2
                                                                            SELECT SUM(f2) FROM tbl1;
                              INSERT INTO tbl1 (f1,f2) VALUES (15,20);
                              COMMIT;
                                                                            SELECT SUM(f2) FROM tbl1;

                    • Serializable - cамый высокий уровень изолированности; транзакции полностью изолируются друг от друга.
                      На этом уровне результаты параллельного выполнения транзакций для базы данных в большинстве случаев можно считать
                      совпадающими с последовательным выполнением тех же транзакций (по очереди в каком-либо порядке). Это самый высокий уровень изоляции,
                      и решает проблему фантомного чтения, заставляя транзакции выполняться в таком порядке, чтобы исключить возможность конфликта.
                      В двух словах, уровень SERIALIZABLE блокирует каждую строку, которую транзакция читает. На этом уровне может возникать множество
                      задержек и конфликтов при блокировках. На практике данный уровень изоляции применяется достаточно редко,
                      но потребности вашего приложения могут заставить вас использовать его, согласившись с меньшей степенью совместного
                      доступа в пользу стабильности данных

    • довговічність - після того, як транзакція завершилася та зафіксувала свої результати в базі даних, система має гарантувати,
                      що ці результати переживуть будь-які подальші збої(виелючили світло тощо...)



✅✅✅  SQL   ✅✅✅

    • реляционной модели Кодда  - должна включать поддержку языка, обеспечивающего манипулирование структурами БД и содержащейся в этих
                                  структурах информацией. Стандартом де-факто такого языка на текущий момент является Structured Query Language - SQL
    • различные объекты         - таблицы, сохраненные процедуры, пользовательские функции, триггеры, представления, правила
    • таблицы можно разделить на два типа:
        • пользовательские
        • системные
    • NULL говорит о том, что значение поля не известно
    • SELECT позволяет выполнять три основные операции
        • проектирование    - определение списка выбираемых полей
        • выбор             - определение списка выбираемых строк
        • объединение       - выбор из нескольких таблиц с описанием взаимосвязей между ними
    • UNION && UNION_ALL    - UNION_ALL - включить дубликати, менее ресурсоемка
    • ORDER BY              - сортировка
    • WHERE                 - BETWEEN, AND, OR, IN, NOT IN, LIKE, NOT LIKE, IS
    • LIKE                  - поиск значений по шаблону
    • Обєдинение таблиц     - ANSI, Transact SQL
        • ANSI              - взаимосвязи между таблицами описываются в предложении FROM
                              SELECT select_list
                              FROM table1 JOIN table2 ON join_conditions1
                              [JOIN table3 ON join_conditions2 ...]
        • Transact SQL      - условия объединения описываются в предложении WHERE
                              SELECT *
                              FROM Clients c, Accounts a
                              WHERE a.ClientID = c.ClientID
    • Декартово произведение
        • сервер выбирает записи из всех таблиц, перечисленных в предложении FROM, комбинирует их по принципу "каждая с каждой",
          но в окончательный resultset включает только те комбинации, которые удовлетворяют условиям объединения
    • Самообъединение - в соответствие одной записи таблицы ставится другая запись из той же таблицы
        SELECT p.spid, p.status, b.spid, b.hostname, b.cmd
        FROM master..sysprocesses p
        JOIN master..sysprocesses b ON b.spid = p.blocked_spid

    🔶 Підзапити
        • вкладення одного запиту в інший
        • при цьому внутрішній запит повертає інформацію, яку використовує зовнішній запит.
        • приклад
                SELECT Name
                FROM Salary
                WHERE Amount > (SELECT Amount FROM Salary WHERE Name = 'Вася')
        • можуть використовувватися в SELECT, FROM, WHERE, JOIN, ORDER BY, GROUP BY, HAVING, INSERT, UPDATE, DELETE,
                SELECT список_колонок,
                	( SELECT список_колонок
                	  FROM список_таблиц
                	  WHERE условия_подзапроса )
                FROM список_таблиц
                WHERE { condition }
                	( SELECT список_колонок
                	  FROM список_таблиц
                	  WHERE условия )
       • Скалярні
            • 1 рядок, 1 колонку
            • SELECT (SELECT MAX(price) FROM products)
       • Row subquery
            • приклад
            SELECT *
            FROM users
            WHERE (first_name, last_name) = (
                SELECT first_name, last_name
                FROM employees
                WHERE id = 1)

       • Table subquery
            SELECT *
            FROM ( SELECT * FROM users) t

       • с точки зрения взаимодействия с внешним запросом различают два основных вида подзапросов: коррелированные и некоррелированные
            • некоррелированные
                • подзапросы выполняются независимо от основного запроса
                • обычно выполняется один раз перед основным
                • используется результат работы для основного
                        SELECT *
                        FROM employees
                        WHERE salary > (
                            SELECT AVG(salary)
                            FROM employees
                        );
            • коррелированные
                • получают данные из внешнего запроса и зачастую могут служить альтернативой объединению таблиц
                • SELECT a.AccountID,
                        ( SELECT c.LastName
                          FROM Clients c
                          WHERE c.ClientID = a.ClientID )
                  FROM Accounts a
                  WHERE a.Currency = 'UAH'
                        • спочатку виконається основний SELECT
                        • потім виконається внутрішній котрий буде використовувати дані із основного


    • EXISTS наличия той или иной информации
        • получения списка клиентов, не открывших ни одного счета
            SELECT *
            FROM Clients c
            WHERE NOT EXISTS( SELECT 1 FROM Accounts a WHERE a.ClientID = c.ClientID )



    🔶 GROUP BY
        • разделяет таблицу на наборы данных
        • понятие группировки неразрывно связано с понятием агрегатных функций
        • агрегатные функции
            • выполняют вычисление на наборе значений и возвращают одиночное значение
            • детерминирование возвращают одну и ту же величину при каждом их вызове на одном и том же наборе входных значений
            • если используется в выборке без GROUP BY, то она применяется ко всем записям выборки, иначе для каждой группы в отдельности
            • используєтся в SELECT, HAVING, Как оконные функции (OVER), подзапросах
        • позволяет разделить таблицу на наборы данных
        • агрегирующие функции вычисляют для каждого набора итоговое значение
        • єти значения называются агрегирующим вектором
        • значение NULL - строки составят в результатах запроса собственную группу
        • приклад:
                        SELECT dbid, sum(size)
                        FROM sysusages                          FROM ~> WHERE ~> GROUP BY ~> SELECT
                        WHERE crdate > "2005-01-01"
                        GROUP BY dbid

        • HAVING & WHERE
            •  WHERE фільтрує рядки до GROUP BY                                 FROM  ~>  WHERE    ~>  GROUP BY  ~>  SELECT
            •  WHERE не можна використовувати агрегати
            •  HAVING фільтрує группи після GROUP BY                            FROM  ~>  GROUP BY  ~>  WHERE    ~>  SELECT
            •  HAVING зазвичай застосовується до агрегатів
                        SELECT    dbid, sum(size)
                        FROM      sysusages
                        GROUP BY  dbid
                        HAVING    sum(size) > 1000
            • приклад комбінування
                        SELECT       dbid, sum(size)
                        FROM         sysusages                                  FROM  ~>  WHERE  ~>  GROUP BY  ~>  HAVING  ~>  SELECT
                        WHERE        crdate > '2005-01-01'
                        GROUP BY     dbid
                        HAVING       sum(size) > 1000;
            • приклад
                        SELECT  dbid, crdate, size
                        FROM    sysusages                                       FROM  ~>  WHERE  ~>  HAVING  ~>  SELECT
                        WHERE   crdate < ‘2005-01-01’
                        HAVING  size > avg(size)


    🔶 Window function
        • функції які виконують обчислення над набором рядків але не схлопують rows як GROUP BY
        • аналізувати сусідні рядки
        • фактично OVER і говорить що це windows function
        • SUM(salary) - це aggregate        |       SUM(salary) OVER (...)
        • логіка
                • визначає partition
                • сортує rows
                • створює “window frame”
                • виконує функцію
                
        • Window functions
            PARTITION BY працює схоже на GROUP BY але не згортає рядки
            GROUP_BY                    many rows -> one row
            PARTITION BY                many rows -> many rows
            Агрегаційні функції

                AVG()                                                              AVG(salary) OVER (PARTITION BY department)
                COUNT(*)                                                           COUNT(*) OVER ()
                MIN/MAX                                                            MAX(salary) OVER (PARTITION BY department)

            Рангові функції
                ROW_NUMBER()                Просто унікальний номер                ROW_NUMBER() OVER (ORDER BY salary DESC)
                RANK()                      Однакові значення → однаковий rank     RANK()       OVER (ORDER BY salary DESC)
                                                                                                           salary      rank
                                                                                                            100          1
                                                                                                            100          1
                                                                                                             90          3

                DENSE_RANK()                Без пропусків                          DENSE_RANK() OVER (ORDER BY salary DESC)
                                                                                                           salary      rank
                                                                                                            100          1
                                                                                                            100          1
                                                                                                             90          2
            Навігаційні функції
                LAG()                       Попередній рядок                       SELECT month, revenue, LAG(revenue, 1) OVER (ORDER BY month) AS prev_revenue FROM sales
                                            можна задавати                                  month       revenue         prev
                                            офсет                                            Jan          100           NULL
                                                                                             Feb          200           100
                                                                                             Mar          300           200
                LEAD()                      Наступний рядок                        LEAD(revenue, 1) OVER (ORDER BY month)
                                            можна задавати                                  month       revenue         prev
                                            офсет                                            Jan          100           200
                                                                                             Feb          200           300
                                                                                             Mar          300           NULL

                • якщо оновити через UPDATE

                • snapshot результату
                • дані НЕ оновлюються автоматично, потрібно:   REFRESH MATERIALIZED VIEW sales_stats

            • рекурсивний view
                • CREATE RECURSIVE VIEW employee_tree AS
                • використовується для
                    • дерева, графи, категорій, organizational hierarchy

                • живе лише в рамках session
                  CREATE TEMP VIEW temp_data AS SELECT ...

            • можна оновлювати коли не має:
                • GROUP BY, aggregates, DISTINCT, UNION, складних JOIN

            • WITH CHECK OPTION
                • обмеження для VIEW, яке не дозволяє INSERT або UPDATE (через view), що не відповідають умовам view
                • якщо view показує тільки “дозволені” рядки, то WITH CHECK OPTION гарантує, що через це view ти не зможеш записати “заборонені” дані
                • не дозволяй через VIEW записати дані, які це VIEW потім не покаже
                • є таблиця:    CREATE TABLE users (id SERIAL, name TEXT, active BOOLEAN)

                                SELECT      *
                                FROM        users
                                WHERE       active = true

                                UPDATE  active_users

                  Щоб цього уникнути потрібно створити view ось так:

                  SELECT *
                  FROM   users
                  WHERE  active = true
                  WITH CHECK OPTION

                • WITH CASCADED CHECK OPTION (дефолт)   - перевіряє всі умови у ланцюжку view

select * from subjects where subject_name != 'Mathematics'; -- same as above. Both "<>" and "!=" are NOT EQUAL TO operator in SQL.
select * from staff_salary where salary > 10000; -- All records where salary is greater than 10000.
select * from staff_salary where salary < 10000; -- All records where salary is less than 10000.


select *
  from staff_salary
  where salary < 10000
  order by salary desc; -- All records where salary is less than 10000 and the output is sorted in descending order of salary.

select *
  from staff_salary
  where salary between 5000 and 10000; -- Fetch all records where salary is between 5000 and 10000.

select *
  from subjects
  where subject_name IN ('Mathematics', 'Science', 'Arts'); -- All records where subjects is either Mathematics, Science or Arts.

select *
  from subjects
  where subject_name NOT IN ('Mathematics', 'Science', 'Arts'); -- All records where subjects is not Mathematics, Science or Arts.

  from subjects

select *

select *
  from staff
  where age > 50 and gender = 'F'; -- Fetch records where staff is female and is over 50 years of age. AND operator fetches result only if the condition mentioned both on left side and right side of AND operator holds true. In OR operator, atleast any one of the conditions needs to hold true to fetch result.

select *
  from staff
  where first_name LIKE 'A%' and last_name LIKE 'S%'; -- Fetch record where first name of staff starts with "A" AND last name starts with "S".

select *
  from staff
  where first_name LIKE 'A%' or last_name LIKE 'S%'; -- Fetch record where first name of staff starts with "A" OR last name starts with "S". Meaning either the first name or the last name condition needs to match for query to return data.

select *

-- JOINS


SQL Joins: There are several types of JOIN but we look at the most commonly used:
  Inner Join
    - Inner joins fetches records when there are matching values in both tables.
  Outer Join
    - Left Join
        - Left join fetches all records from left table and the matching records from right table.
        - The count of the query will be the count of the Left table.
        - Columns which are fetched from right table and do not have a match will be passed as NULL.
    - Right Join
        - Right join fetches all records from right table and the matching records from left table.
        - The count of the query will be the count of the right table.
        - Columns which are fetched from left table and do not have a match will be passed as NULL.
    - Full Join
        - Full join always return the matching and non-matching records from both left and right table.


-- Inner Join: 21 records returned – Inner join always fetches only the matching records present in both right and left table.
-- Inner Join can be represented as either "JOIN" or as "INNER JOIN". Both are correct and mean the same.
select count(1)
  from staff stf
  join staff_salary ss on ss.staff_id = stf.staff_id
  order by 1;


-- 24 records – 24 records in right table.
-- All records from RIGHT table with be fetched irrespective of whether there is matching record in the LEFT table.
select count(1)
  from staff stf
  right join staff_salary ss on ss.staff_id = stf.staff_id
  order by 1;


-- 26 records – all records from both tables. 21 matching records + 2 records from left + 3 from right table.
-- All records from both LEFT and RIGHT table with be fetched irrespective of whether there is matching record in both these tables.
select count(1)
  from staff stf
  full outer join staff_salary ss on ss.staff_id = stf.staff_id
  order by 1;

-- #1 Using JOIN keyword between tables in FROM clause.
select t1.column1 as c1, t1.column2 c2, t2.column3 as c3
  from table1 t1
  join table2 as t2 on t1.c1 = t2.c1 and t1.c2 = t2.c2;


-- #2. Using comma "," between tables in FROM clause.
select t1.c1 as c1, t1.c2 as c2, t2.c3 c3
  from table1 as t1, table2 as t2
  where t1.c1 = t2.c1 and t1.c2 = t2.c2;

-- достигается в первую очередь за счёт того, что индекс имеет структуру, оптимизированную под поиск -  например,
-- сбалансированного дерева.



-- VIEW
In PostgreSQL, представление представляет собой псевдотаблицу. Это означает, что представление не является настоящей таблицей.
Однако мы можем ВЫБРАТЬ ее как обычную таблицу. Єто именованный запрос, который выполняется каждый раз при обращении к нему.
Используется во избежание дублирования кода.
Представление

  - может содержать все или некоторые столбцы таблицы
  - представления можно использовать практически везде, где можно использовать реальную таблицу
  - построение представлений на основе других представлений не редкость
  - помогая обеспечить абстракцию данных и улучшить безопасность данных
  - можете адаптировать представление под ваши текущие потребности
  - представление не является неизменяемым, и в нем не содержится "реальных данных". Это означает, что Вы можете легко изменять его,
    не воздействуя на данные в Вашей базе данных:



    WITH [ CASCADED | LOCAL ] CHECK OPTION
    Этот параметр управляет поведением автоматически обновляемых представлений.
    Если этот параметр указан, команды INSERT , UPDATE и MERGE в представлении будут проверяться, чтобы гарантировать,
    что новые строки удовлетворяют условию, (то есть, новые строки проверяются, чтобы гарантировать, что они видны через представление).
    Если CHECK OPTION не указан, командам INSERT, UPDATE и MERGE в представлении разрешено создавать строки, которые не видны через представление.
    Поддерживаются следующие параметры проверки:




  - для баз данных с объемными вычислениями или сложными соединениями,
    индексированные представления могут значительно ускорить обработку запросов
  - представления упрощают и систематизируют агрегацию данных
  - представления как поддержка устаревшего кода при рефакторинге:
      В процессе рефакторинга баз данных представления находят свое важное применение, успешно скрывая изменения структур,
      при этом функциональность устаревшего кода остается прежней
  - представления дают возможность открывать только части данных, когда приложению требуется лишь фрагмент полной базы данных
  - сокращая необходимость в дублировании запросов, представления сэкономят время разработчика, и нередко способствуют
    сохранению его психического здоровья


      CREATE [OR REPLACE] VIEW view-name AS
        SELECT column(s)
        FROM table(s)
        [WHERE condition(s)];


Пример:
      В представление будут добавлены только записи, цена которых превышает 200.
      CREATE VIEW Price_View AS
        SELECT id, price
        FROM Price
        WHERE price > 200;

      DROP VIEW [IF EXISTS] view-name;



Єсть материализованные представление и обычная вьюха

        CREATE MATERIALIZED VIEW customer_orders AS SELECT customer_id, COUNT(*) as total_orders FROM orders GROUP BY customer_id;

        SELECT * FROM customer_orders;


  Это материализованное представление быстро устареет, пока вы его не обновите: даже если вы добавляете новые данные в базовую таблицу (или обновляете или удаляете данные),
материализованное представление не включает эти изменения автоматически; это снимок на момент создания.
Чтобы обновить материализованное представление, нужно запустить

        REFRESH MATERIALIZED VIEW customer_orders


  Материализованные представления PostgreSQL, основанные на методе материализации, предварительно вычисляют часто
выполняемые запросы и сохраняют результаты в виде таблицы. В отличие от стандартных представлений PostgreSQL,
которые запускают базовый запрос каждый раз при обращении к представлению, материализованные представления сохраняют результат исходного запроса в базе данных.
Самое замечательное в этом то, что вашей базе данных не придется выполнять запрос каждый раз, когда вы его запускаете: результаты уже доступны на диске —
вы получите ответ на свой запрос гораздо быстрее.
  Это отличный способ оптимизировать ответы на запросы, требующие больших ресурсов для вычислений.
Например, запросы, которые могут включать обработку больших объемов данных, агрегаты или множественные соединения.

  Чтобы поддерживать актуальность материализованных представлений, их необходимо обновлять.

  При обновлении материализованного представления запрос пересчитывается для всего набора данных.
На самом деле, когда вы запускаете обновление, старые материализованные данные удаляются, а затем заменяются новыми,
повторно материализованными данными.
